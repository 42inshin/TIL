# SSL 인증서

# CA

  • 인증서는 클라이언트가 접속한 서버가 클라이언트가 의도한 서버가 맞는지 보장하는 역할을 하는데,
  • 이 역할을 민간기업들이 하며 이 기업들을 CA(Certificate authority) 혹은 Root Certificate 라고 부른다.
  • 신뢰성이 엄격하게 공인된 기업들.

# 사설 인증기관

  • 개발, 사적인 목적을 위해 SSL 암호화 기능을 이용하려 한다면 직접 CA 역할 을 수행 할 수 있다.
  • 하지만 공인된 인증서가 아니기 때문에 브라우저에서 빨간색으로 경고 표시를 출력한다.

# SSL 인증서의 내용

  1. 서비스의 정보 (인증서를 발급한 CA, 서비스의 도메인 등등)
  2. 서버 측 공개키 (공개키의 내용, 공개키의 암호화 방법)
  • CA에 의해서 서비스 도메인, 공개키와 같은 정보는 암호화 되어 클라이언트에 보내진다.
  • CA의 비공개키를 이용하여 서버가 제출한 인증서를 암호화하는 것임.
  • 각각의 브라우저는 CA의 리스트를 알고 있으며, 각 CA 의 공개키 또한 이미 알고 있다.

# SSL 인증서가 서비스를 보증하는 방법

  • 서버: 인증서 제공
  • 브라우저: CA리스트 확인
  • 브라우저: 해당 CA 의 공개키를 이용하여 인증서 복호화
  • 복호화 될 경우, 해당 인증서가 CA에 의해서 발급된 것을 인증.
  • 해당 서비스가 신뢰 할 수 있다는 것을 의미.

# SSL 동작방법

  • SSL은 암호화된 데이터를 전송하기 위해 공개키와 대칭키를 혼합해서 사용한다.
  • 클라이언트와 서버가 실제 주고받는 정보는 대칭키 방식으로 암호화 한다.
  • 대칭키 방식으로 암호화된 실제 정보를 복호화할 때 사용할 대칭키는 공개키 방식으로 암호화해서 클라이언트와 서버가 주고받는다.
  • 악수(handshake)
    • 클라이언트가 서버에 접속(Client Hello)
      • 클라이언트 측에서 생성한 랜덤 데이터
      • 클라이언트가 지원하는 암호화 방식들
      • 세션 아이디
    • 서버의 응답(Server Hello)
      • 서버 측에서 생성한 랜덤 데이터
      • 서버가 선택한 클라이언트의 암호화 방식
      • 인증서
    • 양측의 랜덤 데이터로 pre master secret 생성
    • 서버에게 받은 인증키 안의 공개키로 pms 를 암호화
    • 서버는 자신의 비공개키로 pms를 복호화
    • 양쪽 모두 일련의 과정을 거쳐 master secret 값을 만듬
    • master key로 session key를 생성
    • 클라이언트와 서버는 핸드쉐이크 단계 종료를 서로에게 알림
  • 세션
    • 실제로 서버와 클라이언트가 데이터를 주고 받는 단계
    • session key를 대칭키 방식으로 이용해 데이터 주고 받음
  • 세션종료
    • 데이터 전송이 끝나면 SSL 통신이 끝났음을 서로에게 알려준다.
    • 이 때 통신에서 사용한 대칭키인 세션키를 폐기한다.

# 7. 학습 내용에 대한 개인적인 총평

  • ssl 동작 방식에 대해서 자세히 다뤄보았다. key와 인증서가 https 에 필요하다는 것을 알았지만, 클라이언트(브라우저)와 서버 사이에 갖가지의 정보가 교환되어서 key를 만든다는 점이 매우 흥미로웠다. 보안을 높이기 위해서 여러가지 기술력이 웹이라는 공간에 들어 갔다는 점에 놀랐고, 하나의 키를 교환하기 위해서 pre master secret, master secret, session key 순으로 바뀌는 점에 있어서도 신기하였다.
  • 지금 배우고 있는 c 언어 이외에도 배워야 할 것들이 무궁무진하고, 이것을 적용하여 새로운 것들을 만들 수 있는 시점이 올 때까지 배움을 게을리 하지 않도록 노력해야겠다.
