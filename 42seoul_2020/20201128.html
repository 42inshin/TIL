# get_next_line 에 대하여

  • 기본 형태
  int get_next_line(int fd, char **line);

  • 제출 파일

    • get_next_line.c
    • get_next_line_utils.c
    • get_next_line.h
    • get_next_line_bonus.c
    • get_next_line_utils_bonus.c
    • get_next_line_bonus.h

  • 반환 값

    • 1 : 한 줄을 읽었을 때
    • 0 : EOF(End Of File) 에 도달했을 때
    • -1 : 에러 났을 때

  • 사용 가능 외부 함수

    • read, malloc, free

  • 설명
    : 줄 바꿈 없이 파일 설명자에서 읽은 줄을 반환하는 함수를 작성하세요.

  • 필히 알아야 되는 것
    : 정적 변수 개념과 사용법.

  • 보너스 조건

    • 하나의 정적변수로 get_next_line 만들기.
    • get_next_line을 사용하여 다중 파일 디스크립터를 관리 할 수 있는 것.
    • 예를 들어, 파일 디스크립터 3, 4, 5가 읽을 수 있는 경우, get_next_line은 3에서 한 번, 4에서 한 번, 다시 3에서 한 번, 5에서 한 번 호출할 수 있습니다. 각 디스크립터의 리딩 쓰레드를 잃지 않고..

  • 헤더 norminette

    • norm 체크 시 중간에 만든 BUFFER_SIZE 에서 에러가 난다.

    • 처음과 끝의 ifndef, endif 의 경우 # 과 붙여주지만 안쪽에 만들어지는 ifndef, define, endif 의 경우엔 각각을 한칸씩 더 띄워줘야 작동하는 것을 알게됨.

    • norminette 를 위해선 아래와 같이 작성하자!

      #ifndef GET_NEXT_LINE_H
# define GET_NEXT_LINE_H

# include <unistd.h>
# include <stdlib.h>

# ifndef BUFFER_SIZE  // # 뒤에 한 칸 띄운다.
#  define BUFFER_SIZE // # 뒤에 두 칸 띄운다.
# endif               // # 뒤에 한 칸 띄운다.

int	get_next_line(int fd, char **line);

#endif

# 기타 사항 : big sur & vscode terminal issue

  • big sur 업데이트 후 vscode terminal 느려지는 현상 해결 터미널에서 아래의 코드를 실행하고 vscode를 “다시 실행” 시키니 효과가 있음.

    codesign --remove-signature /Applications/Visual\ Studio\ Code.app/Contents/Frameworks/Code\ Helper\ \(Renderer\).app

    출처 : https://github.com/microsoft/vscode/issues/105446

# 7. 학습 내용에 대한 개인적인 총평

  • Gnl 로 들어서면서 학습 진도상황이 현저하게 줄어들었다. 어떤 방식으로 풀어나가야 할지 모르다 보니, 정체되어서 아무것도 하지 못하는 상황이다. 여러 사람들에게 묻고 어떤식으로 진행해야 할지 머리속에 그려나가거나 정리를 하면서 구상해야하는데 지금의 분위기는 혼자 열심히 하다가 잠깐 막히는 부분에 대해서만 질문을 하다보니, 아예 모르겠다고 질문을 던지기에는 뭔가 성의없어 보일 것 같다.
  • 다행히 먼저 나간 분들이 정리를 해두신 것들을 읽어보면서 막막했던 과제에 조금의 빛이 들어오는 기분이다. 다시 gnl pdf 파일부터 찬찬히 훑어보면서 어떤 방식으로 코드를 짜야 될지 고민을 하는 중이다.
  • 빅서에서 계속 vscode 가 늦어지는 현상이 있었는데, 우연히 찾게 된 글에서 해결법을 알게 되었다. 슬랙에 공유를 통해서 많은 사람들이 도움이 되었길 바란다.
